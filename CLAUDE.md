# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

Experimental harness for an Intro to HPC course project comparing two task-based runtimes (StarPU and PaRSEC) under the Chameleon dense linear algebra library, running CPU-only Cholesky (`dpotrf`) and QR (`dgeqrf`) factorizations. The repo contains: the Nix build for Chameleon+runtimes, a Design-of-Experiments (DoE) generator in R, a SLURM runner for the PCAD cluster, and StarVZ-based trace analysis scripts.

Target hardware is the **`poti` partition on PCAD**. This is where the experiments run. It contains a RTX 4070.

## Environment

Everything is built and run through Nix flakes (`flake.nix`). Three dev shells are exposed:

- `nix develop` (default) — analysis tools: `marp-cli`, `chromium`, `starvz` (tools + R package), R with `DoE.base`, `FrF2`, `tidyverse`, etc. Use this for plotting and DoE generation.
- `nix develop .#starpu` — Chameleon built with `CHAMELEON_SCHED=STARPU`. Exposes `chameleon_dtesting` linked to StarPU.
- `nix develop .#parsec` — Chameleon built with `CHAMELEON_SCHED=PARSEC`. Exposes `chameleon_dtesting` linked to PaRSEC.

`direnv` is configured (`.envrc` → `use flake`), so the default shell loads automatically on `cd`.

**On PCAD**, Nix is wrapped: every `nix` command must be prefixed with `nixw` (a shim installed at `$HOME/bin/nixw`). The SLURM scripts already do this.

## Running experiments

The experiment loop is `scripts/run.sh <starpu|parsec>`. It:

1. Reads `$DESIGN_FILE` (CSV produced by the R DoE scripts) with columns `algorithm,n,b,runtime,scheduler,...,rep`.
2. Filters rows matching the runtime arg.
3. For each row, sets the runtime-specific env (`STARPU_SCHED` / `STARPU_NCPU` for StarPU, `PARSEC_MCA_mca_sched` for PaRSEC), optionally enables tracing (`TRACE=1` → FxT for StarPU, `PARSEC_MCA_profile_filename` for PaRSEC), and invokes `chameleon_dtesting -o <algorithm> -n <n> -b <b> -t 24 --nowarmup`.
4. Parses the last `;`-separated result line out of the log to append `time,gflops` to `results.csv`.

Key fixed knobs in `run.sh`: 24 threads, `OPENBLAS_NUM_THREADS=1`, `OMP_NUM_THREADS=1`, `MKL_NUM_THREADS=1`, `STARPU_WORKERS_NOBIND=1`, `--nowarmup`. The `--nowarmup` flag matters — StarPU warmup with `CALIBRATE=1` + `dmda` can hang without a performance model; the runner avoids this by using non-history-based schedulers (`lws`, `ws`) and skipping warmup.

The three SLURM scripts in `slurm/` (`block_size.slurm`, `n_size.slurm`, `traces.slurm`) all do the same thing: `rsync` the submit dir into `$SCRATCH`, run `scripts/run.sh starpu` then `scripts/run.sh parsec` inside `nixw nix develop .#{starpu,parsec}`, and copy results back to `$SLURM_SUBMIT_DIR/data/<jobname>_<jobid>/` via an EXIT trap. They differ only in which DoE CSV they use and whether `TRACE=1` is set.

## Design of Experiments

DoE CSVs are generated by R scripts in `scripts/doe/` (one per SLURM experiment):

```bash
Rscript scripts/doe/doe_n_size.r      # → doe_n_size.csv
Rscript scripts/doe/doe_block_size.r  # → doe_block_size.csv
Rscript scripts/doe/doe_traces.r      # → doe_traces.csv
```

Factors crossed: `algorithm` ∈ {`dpotrf`, `dgeqrf`}, `runtime:scheduler` ∈ {`starpu:lws`, `starpu:ws`, `parsec:lfq`, `parsec:gd`}, and either `n` (multiples of the block size 480) or `iterations` (which sets `b = n/iterations`). `n_size` and `block_size` use 5 replications; `traces` uses 2. The CSV must be placed at the path in `$DESIGN_FILE` before `run.sh` is invoked.

## Analysis

Result CSVs from PCAD land in `data/<jobname>_<jobid>/results.csv`, with per-run trace dirs under `data/<jobname>_<jobid>/runs/<idx>_<runtime>_<sched>_<algo>_n<n>_b<b>_rep<r>/`.

```bash
Rscript scripts/analysis/plot_n_size.r       # → plots/gflops_vs_n_r.{png,pdf}
Rscript scripts/analysis/plot_block_size.r   # → plots/gflops_vs_b_r.{png,pdf}
Rscript scripts/analysis/plot_traces.r       # → plots/trace_panel_st_<runtime>_<sched>.{png,pdf}
```

`plot_traces.r` reads StarPU/PaRSEC traces via `starvz_read()` and renders Space-Time panels. The base data directory and per-run identifiers are hardcoded near the top of the script — edit them when pointing at a new job ID.

### Trace tables for PaRSEC (StarVZ "phase 1", official vs offline)

StarVZ panels read parquet tables that a "phase 1" produces from the raw trace.
For PaRSEC there are two phase-1 paths:

- **Official** (`scripts/analysis/trace_phase1.sh`): `dbp2paje` + `starvz -1 -t`.
  Works for CPU runs (degraded: `Application.JobId/Iteration/GFlop` NA, `variable`
  empty), but **fails on GPU runs** — no `application.parquet` comes out.
- **Offline** (`scripts/analysis/parsec_phase1.sh`): reconstructs the tables
  straight from `cham_*.dot` + `cham_*.prof-*` via `dbp2xml`, **not** using StarVZ
  phase 1. Per run it runs, in order, `parsec_tasks_to_parquet.r` (→ `tasks` +
  `states`, GPU-aware: CPU workers `CPU<N>` + GPU streams `CUDA<dev>_<stream>`),
  `parsec_dag_to_parquet.r` (→ `dag`), `parsec_application_to_parquet.r`
  (→ `application` + `y` + `colors`, the slot the official path can't make on GPU),
  and `parsec_ready_to_parquet.r` (→ `variable`, reconstructed Ready). dbp2xml is
  resolved once and exported as `$DBP2XML`. All times are µs in these tables except
  the synthetic `application.parquet`, which is ms (matching StarPU); `trace_common.r`'s
  `time_scale_to_ms()` keys off `states.parquet` to rescale dag/tasks/states.

With the offline tables present, the stock StarVZ panels run on a PaRSEC GPU trace:

```bash
bash scripts/analysis/parsec_phase1.sh data/<job>/runs/<...>_parsec_*   # build tables
Rscript scripts/analysis/plot_gpu_st.r [--extra] [run_dir ...]          # panel_st + CPB
```

`plot_gpu_st.r` renders the native `panel_st` (CPB overlay on; ABE/idleness off —
heterogeneous CPU+GPU set, async GPU spans) → `plots/gpu_st_<runtime>_<sched>_<algo>`;
`--extra` also emits `panel_imbalance` and `panel_utilheatmap`. Other native panels
already fed by hand from these tables: `panel_kiteration` (`plot_kiteration.r`),
`panel_ready` (`plot_ready.r`).

## Sync to/from PCAD

```bash
bash scripts/pcad/sync_to_pcad.sh    # push code (excludes data/, plots/, slides/, context/)
bash scripts/pcad/get_from_pcad.sh   # pull results back
```

Both require `Host pcad` configured in `~/.ssh/config` (see `docs_pcad/pcad.md`).

## What NOT to commit

Per `.gitignore`: `traces/`, `logs/`, `context/` (reference PDFs/notes), `.claude/`, `.direnv/`. The `data/` and `plots/` directories are tracked because they hold the experimental results that figures are built from.
